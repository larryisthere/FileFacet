## OQ-001：独立分发版本的自动更新方案

- 状态：待确认
- 提出日期：2026-07-15
- 影响范围：产品分发、更新交互、网络与隐私边界、签名和发布流程

### 背景

用户希望应用提供接近 HandBrake 的自动更新提醒：发现新版本后展示当前版本、可用版本和发布说明，并允许用户跳过该版本、稍后提醒或立即安装更新。

当前工程启用了 App Sandbox 和 Hardened Runtime，默认不发起网络请求。工程尚未接入更新框架，也尚未确定正式分发渠道。引入第三方运行时依赖前需要记录架构决策。

### 待确认问题

1. 正式版本采用 Mac App Store 分发、独立分发，或同时提供两种渠道？
2. 独立分发版本是否采用 Sparkle 2 及其标准更新窗口？
3. 更新包、`appcast.xml` 和发布说明由 GitHub Releases、独立静态站点或其他服务托管？
4. 应用何时检查更新：仅由用户手动触发、启动后定期检查，或同时支持两种方式？
5. 首次启用定期检查时，是否由用户明确选择？检查周期采用多长时间？
6. 下载和安装是否始终由用户在更新窗口中确认？是否允许后台预下载？
7. 发布说明使用内嵌 HTML、远程页面或纯文本？是否需要中英文版本？
8. 跳过版本、稍后提醒、下载失败、签名验证失败和安装失败分别采用什么反馈与恢复方式？

### 当前建议方案

若正式版本采用独立分发，建议使用 Sparkle 2：

- 通过 Swift Package Manager 引入 Sparkle。
- 使用 Sparkle 标准更新窗口，提供“跳过此版本”“稍后提醒”和“安装更新”。
- 在应用菜单中提供“检查更新…”。
- 支持周期性版本检查；发现新版本后再显示更新窗口。
- 通过 HTTPS 提供 `appcast.xml`、发布说明和更新包。
- 更新包使用 Developer ID Application 签名，启用 Hardened Runtime，并完成 Apple Notarization 与 stapling。
- 使用 Sparkle EdDSA 签名验证更新来源；私钥仅保存在发布机器或 CI Secret 中。
- 更新检查只传输完成版本判断所需的请求，不上传视频路径、文件名、标签内容、Bookmark 数据或资料库内容。
- Mac App Store 构建交由 App Store 更新机制处理，并在对应构建中关闭 Sparkle 更新入口。

### 预期用户流程

1. 用户通过“检查更新…”主动检查，或应用按照已确认的策略定期检查。
2. 没有新版本时，主动检查会明确提示当前已是最新版本；定期检查保持安静。
3. 发现新版本时，更新窗口展示应用图标、当前版本、可用版本和发布说明。
4. 用户可以跳过当前版本、稍后提醒或开始安装。
5. 用户选择安装后，应用展示下载和验证进度。
6. 下载及签名验证成功后，Sparkle 安装新版本并重新启动应用。
7. 下载、验证或安装失败时，用户看到明确原因，并可重试或退出更新流程。

### 工程与发布约束

- App Sandbox 需要允许更新检查所需的客户端网络连接。
- Sparkle 的更新组件、XPC 服务和主应用必须使用一致且有效的发行签名。
- 正式更新只能从已签名并完成公证的发行包生成。
- `CFBundleShortVersionString` 用于展示版本，`CFBundleVersion` 用于构建版本与升级判断，两者需要纳入发布流程。
- `appcast.xml` 更新应在更新包完成上传和可用性验证后发布，避免用户收到无法下载的版本。
- 更新机制引入网络请求和第三方运行时依赖，需要分别更新隐私边界与架构决策记录。

### 进入实现前的确认条件

- 确认正式分发渠道。
- 确认 Sparkle 2 作为独立分发更新框架。
- 确认版本检查、下载和安装策略。
- 确认更新托管位置与 Release Notes 格式。
- 完成包含主要流程、关键状态和失败反馈的可操作 HTML 原型，并由用户确认交互与 UI 方向。
